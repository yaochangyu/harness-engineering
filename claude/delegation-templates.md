# 派工 Prompt 模板

用法：複製對應模板，填掉所有 `{...}` 空格。**空格沒填完不要送出**——
agent 是全新 context，你沒寫的它一律不知道。
每個模板結尾的「回報限制」區塊照抄，不要刪。

通用參數建議（理由見 model-dispatch.md 第 2 節）：
搜尋→`Explore`；研究/實作/重構→`general-purpose` + `sonnet`；審查/第二意見→`opus`。

---

## 1. 搜尋（找檔案、找定義、找「這功能在哪實作」）

```
subagent_type: Explore
prompt:
在 {repo 根目錄絕對路徑} 尋找：{要找什麼，具體描述，含可能的同義詞/命名慣例}。
背景：{為什麼要找，找到後要拿來做什麼——這決定它回報什麼細節}。
搜尋廣度：{medium / very thorough}。

回報限制：
- 每個命中點一行：`檔案路徑:行號 — 一句話說明它是什麼`。
- 最多列 10 個，按相關性排序；沒找到就明說「沒找到」＋你搜了哪些關鍵字。
- 不要貼程式碼內容。
```

## 2. 實作（新功能、修 bug）

```
subagent_type: general-purpose（.NET Clean Architecture 用 project-architecture-engineer）
model: sonnet
prompt:
目標：{要做什麼}。
動機：{為什麼做、給誰用}。
範圍：只改 {目錄/檔案清單}；**不要**動 {明確排除項}。
已知脈絡：{相關檔案路徑:行號、既有模式的參考位置、要遵守的規則檔路徑}。
先讀 {專案}/.issues/ 下相關檔案，不要重複已失敗的方法。

驗收條件（逐條可機械判定）：
1. {例：`dotnet build` 通過，0 warning}
2. {例：新增測試 X 通過，既有測試不紅}
3. {例：使用者要求的行為 Y 以實跑輸出證明}

回報限制：
- 回報：改了哪些檔案（路徑:行號）、每條驗收條件的通過證據（指令輸出原文節錄）、
  做不到或未驗的項目明列。
- 結論最多 10 行；不要貼整檔內容。
```

## 3. 重構（行為不變的結構調整）

```
subagent_type: general-purpose
model: sonnet
prompt:
目標：把 {對象} 重構為 {目標結構}，參考既有模式 {模式所在檔案路徑:行號}。
不變式（最重要）：對外行為完全不變——{列出可觀察行為：API 回應、測試結果、輸出格式}。
範圍：只改 {清單}；發現範圍外也該改的，記下來回報，不要動手。

驗收條件：
1. 重構前先跑 {測試指令} 記錄基準，重構後結果必須逐項相同。
2. {例：舊的公開介面呼叫點全部更新，grep 舊名稱為 0 筆}
3. 沒有任何測試被修改或弱化（除了純改名）。

回報限制：同實作模板。額外要求：附「基準 vs 重構後」的測試結果對照。
```

## 4. 研究（技術選型、查文件、跨來源整理）

```
subagent_type: general-purpose
model: sonnet（結論將影響架構決策時用 opus）
prompt:
問題：{要回答的具體問題，一句話}。
決策脈絡：{我們要用這答案決定什麼}。
來源優先序：ctx7 查官方文件 → 專案內既有用法（{路徑}）→ 網路搜尋。
每個結論都要標註來源；查不到的明說查不到，禁止用訓練資料的印象充當查證結果。

驗收條件：
1. 回答直接對應問題（不是泛泛介紹）。
2. 每個關鍵主張有來源（URL 或 檔案路徑:行號）。
3. 分成：已知事實／推論／建議。

回報限制：
- 結論 10 行內落地成 {輸出檔路徑}，對話只回：一句話答案＋檔案路徑＋信心程度（高/中/低＋原因）。
```

## 5. 審查（驗收別人的產出、程式碼審查、第二意見）

```
subagent_type: general-purpose
model: opus
prompt:
你是獨立審查者，沒有參與實作。審查對象：{檔案路徑清單 或 git diff 範圍}。
驗收條件：{逐條列出，與派工時相同}。
（不要告訴審查者原作者是誰、花了多少輪——避免順從偏誤。）

任務：
1. 逐條判定驗收條件：通過／不通過，各附證據（檔案路徑:行號 或 實跑輸出）。
2. 找出：規則互相打架、路徑或指令不存在、會被誤讀的模糊語句、弱化的測試。
3. 對每個問題給嚴重度：blocker（不修不能收）／major／minor。

回報限制：
- 只回問題清單（嚴重度排序）＋逐條判定結果；沒問題就明說「逐條通過」。
- 不要動手修改任何檔案；不要客套、不要「整體而言不錯」這類填充語。
```
