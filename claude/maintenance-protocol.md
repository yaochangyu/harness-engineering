# 維護協議：未來的模型怎麼安全地更新這套制度

這套檔案是給未來所有 session（多半是比建立者弱的模型）沿用的。
更新的原則：**寧可少改，不可改壞**。制度檔改壞的代價由之後每一個 session 承擔。

## 1. 權限分級

### 可以自行做（不用問使用者）
- 在專案的 `.issues/` 新增教訓（格式見第 3 節）。
- 在 judgment-rubrics.md / delegation-templates.md **追加**正反例（不改既有判準本身）。
- 修正已驗證失效的路徑或指令：先實際確認新路徑存在／新指令可跑，才能改；
  改完在檔案底部的變更紀錄留一行。
- 修錯字。

### 要先問使用者（提出 diff，等確認）
- 刪除或改寫任何既有規則、判準、門檻數字（例如「重試兩輪」改成三輪）。
- 修改 `~/.claude/CLAUDE.md` 的路由表。
- 新增會**每個 session 都載入**的內容（CLAUDE.md 本體、`~/.claude/rules/` 下的新檔）。
- 新增 agent 定義檔（`~/.claude/agents/*.md`）。

### 絕對不可自行改（連提議都要標高風險）
- `~/.claude/settings.json`、`settings.local.json`（含 hooks、permissions、MCP 設定）。
- `~/.claude/creds/` 下任何東西。
- rules/git.md 的憑證安全段落（安全規則只能由使用者放寬）。
- 刪除 backup/ 目錄下的任何檔案。

## 2. 改檔前的固定動作
1. 備份：`cp {檔案} /mnt/d/lab/harness-engineering/claude/backup/{檔名}.{YYYY-MM-DD}.md`
   （同日多次修改，備份一次即可）。
2. 改完 read-back 驗證（重新讀取、確認改動落地且沒破壞其他段落）。
3. 在該檔案底部「變更紀錄」加一行：`- YYYY-MM-DD：改了什麼、為什麼（session 主題）`。
   沒有變更紀錄段落就建一個。

## 3. 教訓寫回哪裡、用什麼格式

| 教訓類型 | 寫到哪 |
|---|---|
| 專案特定的失敗方法（某指令在此 repo 不能用） | 該專案 `.issues/{功能}.issues.md` |
| 跨專案通用的派工/判斷教訓 | judgment-rubrics.md 追加正反例，或先記到本檔第 6 節的待整理區 |
| 環境事實變更（工具移除、路徑搬家） | 直接修正對應的 HARNESS 檔（走第 1 節權限分級） |

`.issues` 條目格式（一條一段，倒序排列，最新在上）：
```
## YYYY-MM-DD {一句話標題}
- 情境：當時在做什麼
- 失敗的方法：具體指令/步驟＋錯誤訊息原文（節錄）
- 原因：為什麼失敗（不確定就寫「未查明」）
- 改用：最後成功的方法；還沒解就寫「未解」
```

## 4. 精簡門檻（防止制度檔自己變成 token 黑洞）
- 單一 HARNESS 檔超過 **250 行**，或 `.issues` 單檔超過 **15 條** → 停止追加，
  向使用者提議精簡（合併重複、刪除已過時的條目），拿到同意才動。
- 精簡時的判準：保留「判準與正反例」，優先刪「一次性的事件敘述」。
- CLAUDE.md 本體超過 **60 行** → 這是制度退化的警訊（見 letter-to-future-sessions.md），
  提議把新增內容下放到子檔。

## 5. 安裝 Claude 相關工具後的固定動作

任何會動到 `~/.claude/` 的安裝（框架、外掛、Claude Code 升級）做完後，跑：
```
bash /mnt/d/lab/harness-engineering/claude/check-harness.sh
```
它會檢查：入口 symlink 是否被覆蓋、事實來源是否被 append 汙染、
`~/.claude/rules/` 是否冒出新的自動載入檔、制度庫關鍵檔是否齊全。
有問題會印出對應的修復指令，照做後再跑一次確認。
安裝器寫入的內容若有價值：搬到 `HARNESS/rules/` 新子檔＋路由表加一行，不要留在入口檔裡。
