# 模型調度守則

適用對象：主對話的模型（不論是 Haiku、Sonnet 還是 Opus）。
目的：主對話 context 是最稀缺的資源，粗活外包、只留結論，品質靠合約而不是靠聰明。

## 1. 指揮官不下場

主對話遇到下列任一情況，**必須**派 subagent，不要自己動手：

| 觸發條件 | 派什麼 |
|---|---|
| 需要讀 3 個以上檔案才能回答（讀 HARNESS 制度檔/路由檔屬 meta 操作，不計入） | `Explore`（唯讀搜尋） |
| 掃整個 repo / 找「某東西在哪」 | `Explore` |
| 查網頁、跨多個來源的研究 | `general-purpose` |
| 批次修改多個檔案（模式已確定） | `general-purpose`，model 用 `haiku` 或 `sonnet` |
| 實作一個獨立功能 | `general-purpose`（.NET Clean Architecture 用 `project-architecture-engineer`） |
| 設計實作計畫 | `Plan` |
| 問 Claude Code / API / SDK 本身的問題 | `claude-code-guide` |
| 預期單一指令輸出超過 100 行 | 派 agent 執行並摘要，或輸出導到檔案再抓重點 |

主對話**可以**自己做：讀 1–2 個已知路徑的檔案、單一小修改、跑一個短指令、與使用者對話。

例外：任務只碰 1–2 個已知路徑的檔案時，派工的開銷比省下的 context 貴，直接做。

## 2. 本環境實際可用的型號與參數（2026-07-03 查證）

- Agent tool 的 `model` 參數可選：`haiku`、`sonnet`、`opus`（`fable` 只在特殊 session 存在，**不要指定**）。
- 省略 `model` 時，繼承 agent 定義或主對話的模型。
- **effort 無法在派工時指定**：reasoning effort 只能寫在 agent 定義檔
  （`~/.claude/agents/*.md` frontmatter）。若某類工作固定需要高 effort，
  提議使用者新增一個 agent 定義檔，不要假裝有 per-call effort 參數。
- `subagent_type: "fork"` 會繼承主對話完整 context——只用在「需要目前對話全部脈絡」的任務；
  一般派工用 fresh agent，比較便宜也比較不會被既有偏見帶偏。

預設對照：
| 任務 | model |
|---|---|
| 機械性批次套用（模式已驗證過） | `haiku` |
| 一般搜尋、實作、重構、研究 | `sonnet` |
| 架構判斷、難題除錯、第二意見、審查安全相關變更 | `opus` |

## 3. 派工三件套（缺一不可，模板見 delegation-templates.md）

每個派工 prompt 必須包含：
1. **目標與動機**：要做什麼、為什麼要做（agent 是全新 context，不知道你們剛剛聊了什麼）。
2. **驗收條件**：可機械判斷的完成標準（「build 通過且輸出貼回」而不是「確保品質」）。
3. **回報格式**：規定它回什麼、不回什麼。

## 4. 回報合約（寫進每個派工 prompt 的固定尾巴）

> 回報限制：結論最多 10 行；引用程式碼一律用 `檔案路徑:行號`，不要貼大段內容；
> 產出超過 30 行的（報告、程式碼、清單）寫入檔案，只回傳檔案路徑；
> 有做不到或不確定的事，明確列出，不要含糊帶過。

主對話收到回報後：只把「結論＋路徑」留在對話裡，需要細節再開檔案看指定行號。

## 5. 升降級路徑

- `haiku` 做錯 **1 次** → 直接換 `sonnet` 重派，不要修 prompt 再給 haiku 一次機會。
- `sonnet` 在**同一個子任務**連錯 **2 次** → 帶「完整失敗軌跡」（做了什麼、錯誤訊息原文、
  已排除的原因）升級 `opus`。不帶軌跡的升級等於重來，浪費。
- `opus` 解出來的做法若是**可重複的模式**（例如同一種修法要套 20 個檔案）→
  把模式寫成明確步驟，降回 `haiku`/`sonnet` 批次套用。
- **同一件事最多重試兩輪**（含升級）。還是不行 → 停下來，把失敗軌跡整理給使用者，
  附上你建議的下一步。連續失敗後繼續嘗試是最貴的浪費。
- 每次失敗記入該專案的 `.issues/`（格式見 maintenance-protocol.md），重派前先讀。

## 6. 驗證不自驗

**寫的人不驗收自己的產出。** 因為寫錯的人帶著同樣的誤解去驗，會驗出「通過」。

適用門檻：改動 **2 個以上檔案**、或屬**不可逆/對外**動作（部署、發送、刪除）→ 必須派
fresh-context agent 驗收（不用 fork）。單檔小修 → 自己實跑、附輸出原文當證據即可，不必另派。

| 產出類型 | 驗收方式 |
|---|---|
| 檔案/文件 | fresh agent read-back：只給它「驗收條件＋檔案路徑」，要它逐條判定通過/不通過並引用行號 |
| 程式碼 | 跑 build＋測試，輸出原文當證據；沒有測試就實跑一個最小重現場景 |
| 高風險判斷（架構、安全、不可逆操作前的評估） | 第二意見：派另一個 `opus` agent 獨立作答，比對分歧；或多答案評審選優 |

驗收 agent 的 prompt 不要透露「這是誰做的、花了多少力氣」，只給驗收條件，避免順從偏誤。
驗收不通過 → 帶著驗收報告重派原任務（算入重試次數）。
