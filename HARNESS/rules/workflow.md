# 實作工作流（plan.md / .issues / tree.md）

## 觸發條件（先判斷，別對每件事都啟用整套儀式）
| 情境 | 要不要啟用本流程 |
|---|---|
| 使用者要求實作功能，預計修改 2 個以上檔案或多步驟 | 啟用全套 |
| 單檔小修、改錯字、調設定 | 不啟用，直接做，做完回報 |
| 回答問題、分析、查資料 | 不啟用 |
| 使用者說「自動執行」 | 啟用 plan.md 記錄，但不逐步等確認 |

## session 開始時
- 檢查當前目錄是否有 `*.plan.md`；有未完成項目時，詢問使用者是否繼續處理。

## plan.md 流程
1. 實作前先列計畫：使用 [plan-template.md](../plan-template.md) 作為起點，每步驟用表格追蹤並說明為什麼需要。
   - **只用狀態欄（⬜/🟦/✅/⚠️）追蹤進度，不混用核取方塊**
   - 每步驟的「完成條件」內才用核取方塊記錄細節
2. 計畫輸出到當前目錄，**檔案名稱格式**: `{功能名稱(英文)}-{YYYY-MM-DD}.plan.md`
   - 日期用**計畫建立日期**（非完成日期）
   - 封存時無需改檔名或編輯內容，直接移到 `.archive/` 即可
3. 待使用者確認後才實作；每次只做一步，完成後等確認再下一步。
   例外：使用者說「自動執行」時不需逐步等待。
   （這是使用者明令的唯一逐步請示情境，優先於 judgment-rubrics.md §3「不該逐步問」的反例。）
4. 每完成一步（或遇到問題要問使用者時）就更新 plan.md 的狀態欄，讓它永遠反映目前狀態。
5. 所有步驟完成後，**直接移到 `.archive/` 資料夾**（無需編輯內容，檔名已包含日期）
   ```bash
   mv {功能名稱(英文)}-{YYYY-MM-DD}.plan.md .archive/
   git add .archive/{功能名稱(英文)}-{YYYY-MM-DD}.plan.md
   git commit -m "archive: {描述} ({YYYY-MM-DD} completed)"
   ```

**關鍵設計原則**:
- 使用 **單一追蹤機制**：狀態表用 ⬜/🟦/✅/⚠️ 表達進度
- 避免混用核取方塊與狀態欄（易造成混亂）
- 遇到阻塞（使用者決定項）時改為 ⚠️ 狀態，並記在「遭遇的問題」區塊

## .issues 問題紀錄（避免重複踩坑）
- 位置：當前專案的 `.issues/` 資料夾；檔名 `{功能名稱(英文)}.issues.md`。
- 執行計畫發生錯誤或審查出問題時記錄：失敗的方法、步驟、原因。
- **動手前先讀對應的 issues 檔**，不要重複使用已失敗過的方法。
- 格式見 maintenance-protocol.md 的「教訓格式」。

## tree.md 維護
- 專案有 `tree.md` 時：每次新增、刪除、移動檔案或資料夾，都要更新它。
- 不記錄：`.gitignore` 定義的項目、`bin/`、`obj/`。
- 專案沒有 tree.md 時不需要主動建立。

## 完成的最低要求
- 功能實作後一定要 build，build 輸出貼進回報（失敗就是沒完成）。
- 詢問使用者是否需要執行測試；需要就執行。
- 更嚴格的完成判準見 judgment-rubrics.md 第 2 節。
