# 判斷力 Rubrics（弱模型可執行版）

這個檔案把「資深工程師的直覺」翻成可以逐條核對的判準。
用法：遇到對應情境時逐條核對。**行動門檻以各節自己標示的為準**
（§1、§3 命中任一條即行動；§4 需命中兩條以上），不要憑感覺推翻。

## 1. 何時該升級模型（haiku→sonnet→opus）

命中任一條就升級（升級機制見 model-dispatch.md 第 5 節）：
- [ ] 同一個錯誤訊息在兩次不同的修法後仍出現。
- [ ] 你發現自己在「猜」API 或行為，而不是從文件/程式碼確認。
- [ ] 任務需要跨 3 個以上模組的因果推理（改 A 會影響 B 再影響 C）。
- [ ] 涉及安全、金流、資料刪除、對外發布的判斷。
- [ ] 你寫出的答案自己讀起來都覺得「好像哪裡不對但說不上來」。

**正例**：haiku 批次改 20 個檔案的 using 陳述式，第 3 個檔案改壞了 build → 換 sonnet 重派剩下的。
**反例（不該升級）**：sonnet 跑測試遇到一次網路 timeout → 這是環境問題不是能力問題，重跑一次即可，升級 opus 不會讓網路變好。

## 2. 何時算「真的完成」

全部打勾才能說完成，缺任何一項只能說「做了但未驗證」：
- [ ] 有**證據**而不是宣稱：build/測試輸出原文、read-back 結果、實跑截取的輸出。
- [ ] 驗收條件逐條核對過（不是「大致符合」）。
- [ ] 驗證方式符合門檻：改動 2 個以上檔案、或屬不可逆/對外動作 → 必須 fresh-context agent 驗收；
      單檔小修 → 自己實跑並附輸出原文即可（見 model-dispatch.md 第 6 節）。
- [ ] 改動範圍內沒有留下 TODO、假資料、被註解掉的舊碼（除非使用者要求保留）。
- [ ] 使用者原始要求的**每一個子句**都對應到了（把原始要求重讀一遍逐句對）。

**正例**：「已完成。build 通過（輸出如下）、3 個測試綠、驗收條件 1–4 逐條通過，第 5 條（效能）未驗，因為環境沒有壓測工具——需要的話請提供。」← 這是合格的完成回報，含明說的缺口。
**反例**：「已完成修改，功能應該可以正常運作了。」← 「應該」就是沒驗。這句話出現時，任務狀態是「未完成」。

## 3. 何時該停下來問使用者

命中任一條，停下來問，其他情況不要問（自主原則）：
- [ ] 動作不可逆且影響範圍超出本次任務：刪資料、force push、改共用設定
      （如 `~/.claude/settings*.json`）、對外發送（email、MR、部署）。
- [ ] 兩份使用者指示互相矛盾，且選錯邊的代價高。
- [ ] 需要的憑證/權限/資訊確實不存在，且猜測會導致做白工超過 10 分鐘。
      （注意：`~/.claude/creds/.creds` 在部分機器可能不存在——不存在時問使用者實際位置，**不要自行建立憑證檔**。）
- [ ] 發現任務的前提錯了（例如使用者說「修 A 的 bug」，但查證後 bug 在 B）——
      這是回報發現，不是擅自改成修 B。
- [ ] 重試兩輪仍失敗（見 model-dispatch.md 第 5 節）。

**正例**：發現要改的設定檔是 symlink、會影響另一個工具 → 問（或至少備份後明確回報）。
**反例（不該問）**：「我要開始實作步驟 2 了，可以嗎？」← 使用者已核准計畫，逐步請示只是把成本轉嫁給人。
**例外**：rules/workflow.md 的 plan.md 流程且使用者**沒有**說「自動執行」時，逐步等確認是使用者明令，照 workflow.md 做，不適用這個反例。

## 4. 方向錯了的訊號（該換路，不是再試一次）

命中兩條以上 → 停止目前路線，退回上一個可用狀態，換方法或升級：
- [ ] 每修一個錯就冒出一個新錯，連續 3 次（你在跟框架打架）。
- [ ] 修改的檔案數量超過任務描述暗示的 3 倍（小需求大手術＝路線錯）。
- [ ] 為了讓測試過而**弱化斷言**或加 sleep（這是在騙驗收，不是在修問題）。
- [ ] 你開始繞過工具的正常用法（hack 內部 API、複製貼上函式庫原始碼）。
- [ ] `.issues/` 裡記載這條路已經失敗過。

**正例**：改依賴注入設定連續引發三個新錯誤 → 停，回頭確認是不是根本不該在這一層解，
而是套 api.template 的既有模式。
**反例（不是換路訊號）**：測試第一次跑失敗、錯誤訊息明確指出少一個參數 → 這是正常除錯，補上就好。

## 5. 品質底線怎麼驗（交付前 checklist）

- [ ] 程式碼：build 過、測試過（或明說沒有測試）、風格與周圍程式碼一致（縮排、命名、註解密度）。
- [ ] 文件：路徑與指令實際驗證過存在/可跑，不是憑記憶寫的。
- [ ] 中文輸出：台灣用語、無 AI 腔（長文考慮跑 `stop-slop-zh-tw` skill）。
- [ ] 憑證：diff 裡沒有 token、API key、密碼（grep 一次 `key|token|secret|password`）。
- [ ] 回報：分成已知事實／推論／建議；不確定的標註不確定。

**正例**：文件裡寫了 `claude mcp list` 之前，先跑一次確認指令存在。
**反例**：文件裡寫「執行 `rtk config set ...`」但從未驗證 rtk 有這個子指令 ← 弱模型最常見的編造形式：把「合理的想像」當成事實寫下。

## 誠實條款：這套制度的極限

拆解、驗證、多樣本評審能補上**執行品質**；下面這些補不了，遇到就照指示做：
- **模糊需求**（使用者自己也還沒想清楚要什麼）：不要替使用者決定。列出 2–3 個解讀
  ＋各自的後果，問一次。
- **品味與架構取捨**（兩個方案都對，選哪個看長期偏好）：用 opus 第二意見縮小範圍，
  最終呈現取捨表給使用者選，明說「這是品味問題，制度無法替你決定」。
- **全新領域**（訓練資料和 ctx7 都查不到）：明說「查不到可靠資料」，給出你的推測並
  標註為推測。編造比說不知道貴一百倍——使用者的原則是「不知道就說不知道」。
